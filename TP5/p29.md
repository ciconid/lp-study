## Respuesta

### Marco: hardness y polimorfismo

> *"Para los lenguajes con tipado estático, la propiedad de fuertemente tipado está asociada a un grado de dureza alto en las características vinculada a polimorfismo."*
> — 06-SistemaDeTipos (p. 4)

La dureza respecto al polimorfismo se mide por qué tan estrictamente el sistema de tipos controla el uso de entidades polimórficas y cuándo detecta los errores derivados de ese uso.

---

## Parte 1: polimorfismo por inclusión

### Qué implica un sistema duro para el polimorfismo por inclusión

El polimorfismo por inclusión requiere que un objeto de un subtipo pueda reemplazar a un objeto del supertipo en cualquier contexto. Para que esto sea seguro (no genere errores de tipos), el sistema debe poder garantizar la **sustitución**: que los métodos del subtipo son compatibles con los del supertipo.

> *"Para evitar las violaciones de tipos se debe poder sustituir a un objeto de una clase derivada por un objeto de la clase base en cualquier contexto. [...] Lo que hacen básicamente es garantizar que la relación de herencia sea una relación de tipo is-a. Lamentablemente esto se vuelve bastante difícil ya que la mayoría de los lenguajes brindan un constructor adecuado para herencia de software, con lo que eso implica."*
> — 06-SistemaDeTipos (p. 4)

Un sistema duro impone restricciones que garantizan (o al menos verifican en compilación) que la sustitución es válida. Uno blando deja esa verificación para el momento de la ejecución.

---

### Java — sistema duro

Java tiene tipado estático: el compilador conoce el tipo declarado de cada variable y verifica en compilación que los métodos invocados existen en ese tipo.

```java
abstract class Animal { abstract void hablar(); }
class Perro extends Animal { void hablar() { System.out.println("Guau"); } }
class Gato  extends Animal { void hablar() { System.out.println("Miau"); } }

Animal a = new Perro();  // ✅ el compilador verifica que Perro es subtipo de Animal
a.hablar();              // ✅ hablar() existe en Animal → válido en compilación
a.ladrar();              // ❌ Error en compilación: ladrar() no está en Animal
```

El compilador garantiza que, cualquiera que sea el objeto concreto que tenga `a` en runtime, podrá responderse la llamada `a.hablar()` porque Animal la declara. Esto es verificación estática de la sustitución.

Sin embargo, Java **no impone** el Principio de Sustitución de Liskov a nivel del lenguaje: una subclase puede redefinir un método en formas que rompan la compatibilidad semántica, y el compilador no lo detecta. La verificación es de firmas y tipos, no de comportamiento.

**Dureza respecto a polimorfismo por inclusión**: **alta** — los errores de tipo en el uso de referencias polimórficas son detectados en compilación.

---

### Python — sistema blando

Python tiene tipado dinámico: no hay tipos declarados en variables ni parámetros. La resolución de métodos ocurre completamente en tiempo de ejecución.

```python
class CBase:
    def __init__(self):
        self.v1 = 1
        self.v2 = 2
    def get_message(self):
        return (self.v1, self.v2)

class CDerivada(CBase):
    def get_message(self):
        return 123   # retorna int, incompatible con el contrato de CBase

base_obj    = CBase()
derived_obj = CDerivada()

len(base_obj.get_message())     # ⇒ 2        (ok)
len(derived_obj.get_message())  # ⇒ Error    (int no tiene len)
```

> *"Una clase derivada puede definir comportamiento que no es compatible con el definido en su clase ancestro."*
> — Slides - Sistema de tipos - Python y TypeScript (p. 21)

Python permite que una clase derivada redefina métodos retornando tipos completamente distintos, ignorando el contrato establecido por la clase base. El sistema de tipos no impone ninguna restricción sobre la herencia; los errores solo se descubren al ejecutar el código.

Además, no existe una verificación estática de que un objeto usado en un contexto dado sea realmente del tipo esperado: Python simplemente intenta ejecutar la operación y lanza una excepción si falla.

**Dureza respecto a polimorfismo por inclusión**: **baja** — ninguna restricción estática, no se garantiza la sustitución, los errores se detectan en runtime.

---

### Conclusión: polimorfismo por inclusión

**Java tiene el sistema de tipos más duro**.

El motivo central es el tipado estático: el compilador verifica en el momento de compilación que los métodos invocados sobre una referencia polimórfica existen en el tipo declarado, detectando errores antes de la ejecución. Python delega toda esta responsabilidad al runtime, sin ninguna garantía anticipada.

## Parte 2: polimorfismo por sobrecarga

### Java

Java permite sobrecarga de métodos con resolución **estática en compilación**: el compilador determina cuál variante invocar basándose en los tipos declarados de los argumentos. Los errores de ambigüedad o incompatibilidad se detectan en compilación.

```java
void imprimir(int x)    { System.out.println("int"); }
void imprimir(double x) { System.out.println("double"); }

imprimir(5);    // resuelto en compilación → int
imprimir(3.14); // resuelto en compilación → double
// imprimir("hola"); → Error en compilación: no hay imprimir(String)
```

**Dureza**: alta — resolución estática y temprana; los errores de tipo son capturados en compilación.

### Python

Python **no permite sobrecarga de funciones o métodos por nombre**. Si se definen dos funciones con el mismo nombre, la segunda reemplaza a la primera silenciosamente, sin ningún aviso:

```python
def f1(x):     print(x)
def f1(x, y):  print(x)   # silenciosamente reemplaza a la anterior

f1(4)     # ⇒ Error — no existe más la versión de un argumento
```

> *"Python no permite sobrecarga de funciones, pero sí cuenta con operadores sobrecargados."*
> — Slides - Sistema de tipos - Python y TypeScript (p. 18)

La sobrecarga de **operadores** sí existe, pero se resuelve en runtime mediante dunder methods (`__add__`, `__mul__`, etc.) sin ninguna verificación estática.

**Dureza**: baja — no hay sobrecarga de funciones; el reemplazo silencioso no emite ningún error. La sobrecarga de operadores se resuelve en runtime sin type checking estático.

---

### Conclusión: polimorfismo por sobrecarga

**Java también tiene el sistema de tipos más duro**.

La sobrecarga en Java se verifica y resuelve en compilación; Python no tiene sobrecarga de funciones (y el reemplazo silencioso representa ausencia de control) y su sobrecarga de operadores es enteramente dinámica.

---

## Resumen

| Dimensión | Java | Python |
|---|---|---|
| **Polimorfismo por inclusión** | **Más duro** — verificación estática en compilación, el compilador controla el uso de referencias polimórficas | **Más blando** — ninguna restricción estática, los errores de tipo en jerarquías se detectan en runtime |
| **Polimorfismo por sobrecarga** | **Más duro** — sobrecarga con resolución estática; errores de ambigüedad en compilación | **Más blando** — no existe sobrecarga de funciones; operadores sobrecargados con resolución dinámica |

En ambos casos, la diferencia fundamental es la misma: el tipado estático de Java permite que el sistema de tipos imponga restricciones y detecte errores antes de la ejecución, mientras que el tipado dinámico de Python delega todos estos controles al runtime.
