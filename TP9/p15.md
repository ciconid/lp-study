## Pregunta 15
> Una sintaxis alternativa para la expresión condicional en SML puede ser definida como:
> fun nuevoif (cond,e1,e2) = if cond then e1 else e2;
> Explique por qué factorial trabaja mal si es implementado utilizando nuevoif.

---

### Respuesta

La función `nuevoif` definida como `fun nuevoif (cond, e1, e2) = if cond then e1 else e2;` evalúa siempre ambos argumentos `e1` y `e2` antes de llamar a la función, debido a la evaluación estricta de los argumentos en SML. Esto es diferente al comportamiento del `if` nativo, que solo evalúa la rama correspondiente.

Si se implementa factorial usando `nuevoif`, por ejemplo:
```sml
fun fact n = nuevoif(n=0, 1, n*fact(n-1));
```
SML evaluará siempre tanto `1` como `n*fact(n-1)`, incluso cuando `n=0`. Esto provoca que la recursión nunca termine y se produzca un error de stack overflow.

> *"En SML, las funciones evalúan todos sus argumentos antes de ejecutarse (evaluación estricta). Por eso, si una función condicional como nuevoif recibe ramas recursivas, ambas se evalúan siempre, a diferencia del if nativo que solo evalúa la rama necesaria."*
> — Respuesta basada en el modelo y bibliografía funcional estándar

---

#### Información adicional
Este problema es común en lenguajes con evaluación estricta. El `if` nativo de SML es una construcción especial (no una función común) que implementa evaluación condicional, permitiendo que solo se evalúe la rama necesaria.

> *"In strict languages, function arguments are always evaluated before the function is called. This is why user-defined conditionals cannot replace built-in if expressions."*
> — Scott, Programming Language Pragmatics, Chapter 11
