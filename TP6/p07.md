## Pregunta 7
> ¿Qué consideraciones debería contemplar el sistema de tipos cuando se pasan unidades como parámetro?

---

## Respuesta

Cuando una unidad (subprograma) se pasa como parámetro a otra unidad, el sistema de tipos enfrenta un desafío específico: el parámetro formal ya no es un dato simple (un entero, un booleano, etc.) sino algo ejecutable con su propio perfil de tipos. Para garantizar type-safety, el sistema debe contemplar las siguientes consideraciones:

---

### 1. Representar el tipo del subprograma como su protocolo completo

El tipo de un subprograma pasado como parámetro debe incluir toda su **firma** (*protocol*):

- **Número de parámetros** formales.
- **Tipo de cada parámetro** formal (en orden).
- **Tipo de retorno** (para funciones; o la indicación de que es un procedimiento, sin valor de retorno).

Solo así es posible verificar estáticamente si el subprograma que se pasa como argumento es compatible con el parámetro formal que lo recibe.

> *"First, there is the matter of type checking the parameters of the activations of the subprogram that was passed as a parameter. In C and C++, functions cannot be passed as parameters, but pointers to functions can. The type of a pointer to a function includes the function's protocol. Because the protocol includes all parameter types, such parameters can be completely type checked."*
> — Sebesta, *Concepts of Programming Languages* E12 (p. 646)

Por ejemplo, en C++ se declara un puntero a función incluyendo su protocolo completo:

```cpp
float (*pfun)(float, int);
// pfun puede apuntar a cualquier función que tome (float, int) y devuelva float
```

Esto permite que el compilador verifique que el subprograma real pasado como argumento tenga exactamente ese protocolo.

---

### 2. Verificar tipos de parámetros actuales vs. formales en los sitios de llamada

Cuando el subprograma recibido como parámetro es **invocado** dentro del cuerpo de la unidad receptora, el sistema de tipos debe verificar que los argumentos pasados en esa invocación sean compatibles con los parámetros formales del subprograma recibido. Esto es posible solo si el tipo del parámetro formal ya contiene esa información de protocolo.

> *"It is now widely accepted that software reliability demands that the types of actual parameters be checked for consistency with the types of the corresponding formal parameters. Without such type checking, small typographical errors can lead to program errors that may be difficult to diagnose because they are not detected by the compiler or the run-time system."*
> — Sebesta, *Concepts of Programming Languages* E12 (p. 636)

---

### 3. Compatibilidad estructural del protocolo

Al pasar un subprograma como argumento, el sistema de tipos debe definir cuándo dos protocolos son **compatibles**. La regla más común es la **equivalencia estructural**: los tipos de los parámetros y el tipo de retorno deben coincidir exactamente (o cumplir las reglas de coerción del lenguaje) en cada posición.

---

### 4. Consecuencias de omitir estas consideraciones: el caso Pascal

Un ejemplo histórico ilustra el costo de ignorar estas consideraciones. La definición original de Pascal permitía pasar subprogramas como parámetros *sin incluir información de tipos de sus parámetros*. Como resultado:

- Sin compilación independiente, verificar la consistencia de parámetros era posible pero muy complejo, y generalmente no se hacía.
- Con compilación independiente, era directamente imposible verificar incluso el número correcto de parámetros.

> *"The original definition of Pascal (Jensen and Wirth, 1974) allowed subprograms to be passed as parameters without including their parameter type information. If independent compilation is possible (it was not possible in the original Pascal), the compiler is not even allowed to check for the correct number of parameters. In the absence of independent compilation, checking for parameter consistency is possible but is a very complex task, and it usually is not done."*
> — Sebesta, *Concepts of Programming Languages* E12 (p. 648)

---

### Resumen

| Consideración | Descripción |
|---------------|-------------|
| **Protocolo como tipo** | El tipo del subprograma-parámetro debe incluir tipos de parámetros y tipo de retorno |
| **Verificación en sitios de llamada** | Al invocar la unidad recibida, verificar que los argumentos sean compatibles con su protocolo |
| **Compatibilidad estructural** | Definir cuándo dos protocolos son equivalentes/compatibles |
| **Completitud del tipo** | No omitir información de tipo (lección del Pascal original) |

Estas consideraciones son análogas al chequeo de tipos de datos escalares, pero aplicadas a objetos de tipo "función" o "procedimiento": para que el sistema de tipos pueda operar correctamente, la unidad debe ser tratada como un valor de primer orden con un tipo bien definido y completo.
