## Pregunta 19
> Compare la forma en que Smalltalk modela los tipos que son considerados primitivos en Java.

---

### Tipos primitivos en Java

Java distingue explícitamente entre **tipos primitivos** y **objetos**. Los tipos primitivos son:

| Tipo primitivo | Wrapper class |
|---|---|
| `int` | `Integer` |
| `long` | `Long` |
| `double` | `Double` |
| `float` | `Float` |
| `boolean` | `Boolean` |
| `char` | `Character` |
| `byte` | `Byte` |
| `short` | `Short` |

Los primitivos **no son objetos**: no tienen métodos, no participan en la jerarquía de clases, y su representación en memoria es directamente el valor (eficiente, pero fuera del modelo OO). Para tratarlos como objetos se usan las *wrapper classes* (`Integer`, `Boolean`, etc.), y desde Java 5 el mecanismo de *autoboxing/unboxing* convierte automáticamente entre primitivos y sus wrappers.

Esta distinción **rompe la pureza del modelo orientado a objetos**: en Java no *todo* es un objeto.

> *"Típicamente en este grupo entran los tipos numéricos, carácter y booleanos."*
> — 05-Tipos.pdf (p. 1)

---

### Modelado en Smalltalk: todo es un objeto

En Smalltalk **no existen tipos primitivos**. Todo lo que en Java sería un tipo primitivo es una **clase de primera clase** dentro de la jerarquía de objetos, con la misma raíz `Object`.

> *"Todo en el lenguaje es un objeto, desde una constante numérica hasta un sistema complejo, y toda computación se realiza a través de mensajes entre objetos."*
> — Slides - POO y Smalltalk.pdf (p. 13)

---

### Tipos numéricos

En Java: `int`, `long`, `double`, `float` son primitivos. En Smalltalk la jerarquía es:

```
Object
  └── Magnitude
        └── Number
              ├── Integer
              │     ├── SmallInt      (enteros que caben en una palabra de máquina)
              │     └── LargeInteger  (enteros de precisión arbitraria)
              ├── Float
              └── Fraction            (cociente exacto de dos enteros)
```

Cada una de estas clases es un TDA con su propio protocolo de mensajes. Por ejemplo:

```smalltalk
3 factorial          "Integer recibe el mensaje factorial"
3.14 truncated       "Float recibe el mensaje truncated"
(1/3) + (1/6)        "Fraction: retorna 1/2 exacto"
```

Una ventaja notable de Smalltalk es que **no hay overflow de enteros**: cuando un `SmallInt` supera el rango de la palabra de máquina, el sistema automáticamente promueve el objeto a `LargeInteger`, de precisión arbitraria. En Java, un `int` desborda silenciosamente; para precisión arbitraria se necesita usar explícitamente `BigInteger`.

```smalltalk
100 factorial   "retorna el resultado exacto (un LargeInteger de 158 dígitos)"
```

En Java esto requeriría:
```java
BigInteger.valueOf(100).factorial(); // no existe, requiere implementación manual
```

Además, las operaciones aritméticas en Smalltalk son **mensajes** enviados a objetos numéricos:
- `3 + 4` envía el mensaje `+` al objeto `3` con argumento `4`.
- `5 > 2` envía el mensaje `>` al objeto `5`.

En Java, `+`, `*`, `>` son **operadores del lenguaje**, no mensajes, y solo están definidos para los tipos primitivos (o sobrecargados para `String`).

---

### Tipo booleano

En Java: `boolean` es primitivo. Solo puede valer `true` o `false` (literales del lenguaje). `Boolean` es su wrapper.

En Smalltalk, la jerarquía es:

```
Object
  └── Boolean
        ├── True    (singleton: el objeto true)
        └── False   (singleton: el objeto false)
```

`true` y `false` son los únicos objetos de sus respectivas clases (`True` y `False`). Las estructuras de control como `ifTrue:ifFalse:` son **mensajes** enviados a estos objetos, no palabras reservadas del lenguaje:

```smalltalk
"Clase True"
ifTrue: bloqueTrue ifFalse: bloqueFalse [
    ^bloqueTrue value
]

"Clase False"
ifTrue: bloqueTrue ifFalse: bloqueFalse [
    ^bloqueFalse value
]
```

Esto significa que la selección condicional es simplemente polimorfismo: el objeto receptor (`true` o `false`) decide qué bloque evaluar.

> *"Las estructuras de selección en Smalltalk se modelan mediante mensajes a objetos."*
> — Slides - POO y Smalltalk.pdf (p. 20)

En Java, `if`, `while`, `&&` son construcciones sintácticas del lenguaje, no mensajes. La unificación que logra Smalltalk —tratar la selección y la iteración como mensajes polimórficos— es un rasgo que Java no puede reproducir con sus tipos primitivos.

---

### Tipo carácter

En Java: `char` es un primitivo de 16 bits (Unicode). `Character` es su wrapper.

En Smalltalk: los caracteres son instancias de la clase `Character`. Los literales se escriben con el prefijo `$`:

```smalltalk
$A class          "→ Character"
$A isUppercase    "→ true"
$A asciiValue     "→ 65"
$a asUppercase    "→ $A"
```

Toda operación sobre un carácter es un **mensaje** a un objeto `Character`.

---

### String

En Java: `String` es una clase (no primitivo), pero recibe un tratamiento especial por el lenguaje (literales con `"..."`, operador `+` sobrecargado, *string pool*).

En Smalltalk: `String` es simplemente una clase en la jerarquía. Los literales se escriben con comillas simples `'...'`, y todas las operaciones son mensajes:

```smalltalk
'hola' size          "→ 4"
'hola' , ' mundo'    "→ 'hola mundo'"
'hola' at: 2         "→ $o"
```

No hay ningún tratamiento especial del lenguaje; `String` se comporta exactamente como cualquier otra clase.

---

### Tabla comparativa

| Concepto Java | Tipo primitivo Java | Modelo en Smalltalk |
|---|---|---|
| Número entero | `int`, `long`, `short`, `byte` | Clases `Integer` → `SmallInt` / `LargeInteger` (precisión arbitraria) |
| Número real | `double`, `float` | Clase `Float`; también `Fraction` para exactitud |
| Booleano | `boolean` | Clases `True` y `False` (subclases de `Boolean`); singletons |
| Carácter | `char` | Clase `Character`; literal `$x` |
| Cadena | `String` (clase especial) | Clase `String` (sin tratamiento especial) |
| Operaciones aritméticas | Operadores del lenguaje (`+`, `*`, etc.) | Mensajes entre objetos (`3 + 4` = mensaje `+` al objeto `3`) |
| Estructuras de control | Palabras reservadas (`if`, `while`) | Mensajes a objetos booleanos y bloques |
| Overflow numérico | Silencioso (en `int`/`long`) | Promoción automática a `LargeInteger` |
| Wrappers necesarios | Sí (`Integer`, `Boolean`, etc. + autoboxing) | No aplica: ya son objetos de primera clase |

---

### Conclusión

La diferencia fundamental es que Java **saca los tipos primitivos fuera del modelo de objetos** por razones de eficiencia, lo que obliga a mantener dos mundos paralelos (primitivos y wrappers) y mecanismos de conversión como el autoboxing. Smalltalk, en cambio, **incorpora estos tipos directamente en la jerarquía de clases** de forma consistente: son objetos como cualquier otro, pueden recibir mensajes, pueden subclasificarse, y las operaciones sobre ellos son simplemente polimorfismo. Esto logra una mayor consistencia conceptual a costo de algo de eficiencia en tiempo de ejecución.
