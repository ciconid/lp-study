## Pregunta 31
> Java y Python utilizan dos enfoques muy distintos para realizar el chequeo de tipos. Investigue cómo funciona el chequeo de tipos en cada lenguaje. En Java, diferencie el comportamiento sobre tipos primitivos y sobre objetos, considerando el rol de la herencia y el polimorfismo. En Python, investigue cómo se determina si una operación sobre un objeto es válida. ¿En qué momento se detectan los errores de tipo en cada lenguaje?

## Respuesta

### Contexto general

> *"Type checking is simply testing for type errors in a given program, either by the compiler or during program execution. Type checking is an important factor in language reliability. Because run-time type checking is expensive, compile-time type checking is more desirable. Furthermore, the earlier errors in programs are detected, the less expensive it is to make the required repairs."*
> — Sebesta, Concepts of Programming Languages (p. 51)

> *"Tipo y tiempo de chequeo: el chequeo de tipo consiste en verificar que el tipo de una entidad corresponda al esperado por el contexto."*
> — 06-SistemaDeTipos (p. 1)

El momento en que se realizan los chequeos es una consecuencia directa del tipo de ligadura al tipo: estática (Java) o dinámica (Python).

## Java — chequeo de tipos estático

Java tiene tipado **estático**: el tipo de cada variable, parámetro y expresión se conoce en tiempo de compilación. El compilador verifica la coherencia de tipos antes de generar el bytecode.

> *"The design of Java requires checks of the types of nearly all variables and expressions at compile time. This virtually eliminates type errors at run time in Java programs."*
> — Sebesta, Concepts of Programming Languages (p. 51)

### Tipos primitivos (`int`, `double`, `char`, `boolean`, etc.)

Para los tipos primitivos, el chequeo es completamente estático:

- El compilador conoce el tipo de cada variable primitiva declarada.
- Las operaciones entre tipos incompatibles son rechazadas en compilación.
- Las conversiones implícitas (widening) permitidas (`int → long → float → double`) son insertadas automáticamente por el compilador.
- Las conversiones reductoras (`double → int`) requieren un cast explícito; si no se escribe, es error de compilación.

```java
int a = 5;
double b = a;      // ✅ widening implícito (int → double), resuelto en compilación
int c = b;         // ❌ Error en compilación: pérdida de precisión
int d = (int) b;   // ✅ cast explícito
```

No hay chequeo de tipos en runtime para operaciones sobre primitivos: el compilador ya garantizó la corrección.

### Tipos de referencia (objetos) — chequeo mayormente estático

Para objetos, el compilador chequea contra el **tipo declarado** de la referencia, no contra el tipo del objeto en runtime:

```java
Animal a = new Perro();   // ✅ Perro es subtipo de Animal
a.hablar();               // ✅ hablar() existe en Animal → resuelto en compilación
a.buscarHueso();          // ❌ Error en compilación: buscarHueso() no está en Animal
```

El compilador no puede saber el tipo concreto del objeto en runtime (puede ser `Perro`, `Gato`, etc.); solo sabe que es un `Animal`. Por eso solo permite operaciones declaradas en el tipo estático de la referencia. La resolución de qué método concreto ejecutar (**dynamic dispatch**) ocurre en runtime, pero dentro de límites que el compilador ya verificó como seguros.

### Herencia, polimorfismo y chequeos en runtime

Cuando se combina herencia con polimorfismo, algunos chequeos no pueden hacerse en compilación:

#### Upcasting (siempre seguro — verificado en compilación)

```java
Animal a = new Perro();  // Siempre válido: Perro IS-A Animal
```

El compilador acepta esto sin emitir advertencias.

#### Downcasting (requiere chequeo en runtime)

```java
Animal a = new Perro();
Perro p = (Perro) a;   // ✅ correcto en runtime: a SÍ es un Perro
Gato g  = (Gato) a;    // ❌ ClassCastException en RUNTIME
```

El compilador no puede determinar en general si un downcast es válido; delega el control a la JVM. Si el cast es incorrecto, la JVM lanza `ClassCastException` en el momento de la ejecución. Esto es uno de los pocos casos donde Java necesita chequeo en runtime.

El operador `instanceof` permite verificar antes de castear:
```java
if (a instanceof Perro) {
    Perro p = (Perro) a;  // ahora es seguro
}
// Java 16+ pattern matching:
if (a instanceof Perro p) { p.buscarHueso(); }
```

#### Generics y type erasure

Los genéricos (`List<Integer>`, `Map<String, Integer>`) son verificados **completamente en compilación**. En runtime, los parámetros de tipo son borrados (*type erasure*): `List<Integer>` y `List<String>` son idénticos como bytecode. Esto significa que ciertos errores con generics que escapan la verificación estática solo se detectan en runtime.

#### Otros casos de runtime en Java

- `NullPointerException`: invocar un método sobre una referencia `null` — el compilador no puede detectar todos los casos.
- `ArrayStoreException`: asignar a un arreglo un elemento de tipo incompatible con el tipo del arreglo.

### Momento de detección de errores en Java

| Tipo de error | Detección |
|---|---|
| Asignación de tipo incompatible | Compilación |
| Método no existente en el tipo declarado | Compilación |
| Argumentos de tipo incorrecto en llamada | Compilación |
| Conversión reductora sin cast explícito | Compilación |
| Cast inválido entre tipos de referencia | Runtime (`ClassCastException`) |
| Acceso a referencia `null` | Runtime (`NullPointerException`) |
| Error en tipos con generics (escapando type erasure) | Runtime |

## Python — chequeo de tipos dinámico

Python tiene tipado **dinámico**: los tipos no se declaran; se asignan en runtime cuando los objetos son creados y asignados a variables. El intérprete verifica los tipos en el momento exacto en que cada operación se ejecuta.

> *"Los chequeos de tipos se realizan en tiempo de ejecución."*
> — Slides - Sistema de tipos - Python y TypeScript (p. 5)

### Cómo Python determina si una operación es válida: duck typing

Python no verifica el tipo nominal del objeto; verifica si el objeto tiene el **atributo o método** requerido en el momento de la operación. Esto se conoce como **duck typing**.

> *"Python utiliza el concepto de duck typing para realizar parte de los chequeos de tipos. Al usar duck typing, los atributos y métodos de un objeto resultan más relevantes que el tipo del objeto."*
> — Slides - Sistema de tipos - Python y TypeScript (p. 22)

El mecanismo concreto:

1. **Para operadores** (`+`, `%`, `==`, etc.): Python invoca el método especial (*dunder method*) correspondiente del objeto (`__add__`, `__mod__`, `__eq__`, ...). Si el objeto no tiene ese método, se lanza `TypeError`.

```python
x = 5
y = 3
x + y        # Python llama x.__add__(y) → int.__add__(5, 3) → 8 ✅
"hola" + 5   # str.__add__("hola", 5) → NotImplemented
             # int.__radd__(5, "hola") → NotImplemented
             # → TypeError: can only concatenate str (not "int") to str ❌
```

2. **Para acceso a atributos y métodos** (`obj.method()`): Python busca `method` en el `__dict__` del objeto y luego en el MRO (*Method Resolution Order*) de su clase. Si no encuentra el atributo, lanza `AttributeError`.

```python
class Pato:
    def hablar(self): print("Quack!")

class Computadora:
    def encender(self): print("Beep!")

def hacer_hablar(x):
    x.hablar()          # Python busca hablar en x en runtime

hacer_hablar(Pato())         # ✅ Pato tiene hablar
hacer_hablar(Computadora())  # ❌ AttributeError: 'Computadora' object has no attribute 'hablar'
```

> *"Al usar duck typing, los atributos y métodos de un objeto resultan más relevantes que el tipo del objeto."*
> — Slides - Sistema de tipos - Python y TypeScript (p. 22)

3. **Fuertemente tipado a pesar del tipado dinámico**: Python NO aplica coerciones implícitas entre tipos incompatibles. El error se lanza inmediatamente sin intentar convertir los operandos.

```python
"5" + 1    # ❌ TypeError: can only concatenate str (not "int") to str
           # Python no intenta convertir 1 a "1" ni "5" a 5
```

4. **Errores en código no ejecutado**: Un error de tipos en una rama de código que nunca se ejecuta no es detectado hasta que esa rama finalmente se ejecuta.

```python
def f(x):
    if x > 0:
        return x + 1
    else:
        return x + "error"   # ← error de tipos: no se detecta hasta que se llame f(n) con n <= 0

f(5)    # ✅ — la rama con error nunca se ejecuta
f(-1)   # ❌ TypeError — detectado solo ahora
```

> *"Ante un error de tipos, Python lanzará una excepción correspondiente."*
> — Slides - Sistema de tipos - Python y TypeScript (p. 24)

### Momento de detección de errores en Python

| Tipo de error | Detección |
|---|---|
| Operación sobre tipo que no la soporta | Runtime (`TypeError`) |
| Acceso a atributo o método inexistente | Runtime (`AttributeError`) |
| Valor incorrecto de un tipo válido | Runtime (`ValueError`) |
| Errores en código no ejecutado | Al ejecutar esa rama (puede no detectarse nunca en pruebas) |

## Comparación final

| Aspecto | Java | Python |
|---|---|---|
| Tipo de tipado | Estático | Dinámico |
| Momento del chequeo principal | Compilación | Runtime |
| Base del chequeo | Tipo declarado de la variable/referencia | Presencia del atributo/método en el objeto |
| Tipos primitivos | Chequeo 100% estático | No existen tipos primitivos |
| Objetos | Mayormente estático; downcast en runtime | Todo en runtime (duck typing) |
| Herencia y polimorfismo | Upcast estático; downcast + dispatch en runtime | Dispatch en runtime; sin restricciones estáticas |
| Detección de errores | Temprana (en compilación para la mayoría) | Tardía (solo al ejecutar la línea con error) |
| Impacto en confiabilidad | Alta — errores detectados antes de ejecutar | Menor — errores pueden esconderse en ramas no ejecutadas |
| Impacto en eficiencia | Alta — sin overhead de chequeo en runtime para primitivos | Menor — overhead de lookup de atributos en cada operación |
