## Pregunta 30
> Analice, en lo que se refiere al sistema de tipos, la función `módulos` desarrollada en Python y el método `módulos` en Java que se muestran debajo. Indique explícitamente qué características del sistema de tipos son relevantes y en qué grado de dureza están consideradas en cada lenguaje.

### Código del enunciado

**Python:**
```python
def módulos(lista):
    output = []
    for item in lista:
        output.append(item % 2)
    return output
```

**Java:**
```java
public List<Integer> módulos(List<Integer> lista) {
    List<Integer> output = new ArrayList<>();
    for (Integer item : lista) {
        output.add(item % 2);
    }
    return output;
}
```

---

## Respuesta

### Características del sistema de tipos relevantes

#### 1. Tipado estático vs. dinámico

La diferencia central entre ambas implementaciones es el momento en que se conocen y verifican los tipos.

**Python**: Tipado **dinámico**. Ni el parámetro `lista`, ni la variable local `output`, ni el iterador `item`, ni el tipo de retorno tienen un tipo declarado. El intérprete descubre los tipos en tiempo de ejecución.

**Java**: Tipado **estático**. Todos los tipos están declarados explícitamente: `List<Integer>` como tipo del parámetro, `List<Integer>` como tipo de retorno, `List<Integer>` para `output`, e `Integer` para `item` en el ciclo. El compilador verifica la coherencia de todos estos tipos antes de ejecutar el programa.

| Aspecto | Python | Java |
|---|---|---|
| Tipo del parámetro | sin declarar (`lista`) | `List<Integer>` |
| Tipo de `item` | sin declarar | `Integer` (declarado en el for-each) |
| Tipo de retorno | sin declarar | `List<Integer>` |
| Tipo de `output` | lista vacía sin tipo de elemento | `List<Integer>` (parametrizada) |

**Dureza**: Python — baja (sin verificación estática); Java — alta (todo verificado en compilación).

---

#### 2. Declaraciones de tipo y anotaciones

**Python**: No hay ninguna declaración de tipo en la firma de la función. El sistema de tipos no impone ninguna restricción sobre qué tipo puede pasarse como `lista` ni sobre qué devuelve la función. Esto es correcto desde el punto de vista del lenguaje: cualquier iterable cuyos elementos soporten el operador `%` es válido.

```python
módulos([1, 2, 3])       # ✅ funciona
módulos([1.5, 2.7, 3.1]) # ✅ funciona (float soporta %)
módulos(["a", "b"])      # ✅ sintácticamente válido, falla en runtime con TypeError
```

**Java**: La declaración `List<Integer>` restringe tanto la entrada como la salida. El compilador rechaza en tiempo de compilación cualquier llamada que no provea un `List<Integer>`.

```java
módulos(new ArrayList<Integer>())  // ✅
módulos(new ArrayList<Double>())   // ❌ Error en compilación
módulos(List.of(1.5, 2.7))        // ❌ Error en compilación
```

**Dureza**: Python — mínima (no hay restricción de tipos en la firma); Java — máxima en la firma (el compilador rechaza cualquier tipo incompatible con la declaración).

---

#### 3. Polimorfismo paramétrico (generics)

**Python**: No usa polimorfismo paramétrico. La lista `output = []` puede contener cualquier objeto; Python no distingue entre `List<Integer>` y `List<String>` en tiempo de ejecución.

**Java**: Usa **generics** — una forma de polimorfismo paramétrico. `List<Integer>` parametriza la lista para que solo acepte objetos de tipo `Integer`. El compilador verifica:
- Que los elementos agregados a `output` sean compatibles con `Integer`.
- Que el valor retornado sea un `List<Integer>`.

```java
List<Integer> output = new ArrayList<>();
output.add(item % 2);  // item es Integer → item % 2 es int → autoboxing a Integer ✅
output.add("hola");    // ❌ Error en compilación: String incompatible con Integer
```

> *"C++, Java 5.0+, C# 2005+, and F# provide a kind of compile-time parametric polymorphism."*
> — Sebesta, Concepts of Programming Languages (p. 656)

**Dureza**: Python — no aplica (sin generics, sin verificación de tipo de elemento); Java — alta (el tipo del elemento es verificado en compilación por el sistema de generics).

---

#### 4. Duck typing (Python) vs. tipado nominal (Java)

**Python**: El operador `%` sobre `item` es válido para cualquier objeto que implemente `__mod__`. No importa el tipo nominal del objeto; importa que tenga el método requerido. Esto es **duck typing**:

```python
módulos([7, 4, 9])       # item es int → item % 2 resuelto en runtime → ok
módulos([7.5, 4.2, 9.9]) # item es float → item % 2 también válido → ok
módulos(["a", "b"])      # item es str → str no tiene __mod__ con int → TypeError en runtime
```

> *"Python utiliza el concepto de duck typing para realizar parte de los chequeos de tipos. Al usar duck typing, los atributos y métodos de un objeto resultan más relevantes que el tipo del objeto."*
> — Slides - Sistema de tipos - Python y TypeScript (p. 22)

**Java**: `item` es declarado de tipo `Integer`. El compilador sabe exactamente que `Integer` soporta el operador `%` (mediante unboxing a `int`). No hay duck typing; la validez es determinada por el tipo nominal declarado.

**Dureza**: Python — baja (verificación estructural/duck typing en runtime); Java — alta (verificación nominal estática en compilación).

---

#### 5. Coerción implícita: autoboxing/unboxing en Java

En Java, el operador `%` trabaja sobre tipos primitivos (`int`, `long`, etc.), no sobre objetos `Integer`. En la expresión `item % 2`:
- `item` (tipo `Integer`, objeto) es **unboxed** automáticamente a `int` para realizar la operación.
- El resultado `int` es **autoboxed** de vuelta a `Integer` cuando se pasa a `output.add(...)`.

```java
Integer item = 7;
output.add(item % 2);
// equivalente a:
output.add(Integer.valueOf(item.intValue() % 2));
```

Esta coerción implícita es una **reducción menor de la dureza** en Java: el programador no explicita la conversión `Integer ↔ int`. Si `item` fuera `null`, el unboxing lanzaría `NullPointerException` en runtime — un error de tipos que escapa al compilador.

**Python**: No hay coerción relevante en este contexto. El operador `%` se resuelve directamente sobre el tipo del objeto en runtime sin conversiones implícitas.

**Dureza**: Java — ligeramente reducida por autoboxing (hay una conversión implícita no verificada para el caso `null`).

---

#### 6. Inferencia de tipos

**Python**: El tipo de `item` se infiere en runtime a partir de los elementos de `lista`. No hay inferencia estática.

**Java**: El tipo de `item` se infiere del tipo del iterable declarado (`List<Integer>` → `item` es `Integer`) en compilación. Además, el operador diamante `<>` en `new ArrayList<>()` permite al compilador inferir el tipo genérico `Integer` a partir del tipo declarado de `output`.

**Dureza**: Python — inferencia dinámica (en runtime); Java — inferencia estática parcial (en compilación, por el compilador).

---

### Resumen de características por lenguaje

| Característica del sistema de tipos | Python | Dureza Python | Java | Dureza Java |
|---|---|---|---|---|
| Tipado | Dinámico | Baja | Estático | Alta |
| Tipos de parámetros | Sin declarar | Baja | `List<Integer>` explícito | Alta |
| Tipo de retorno | Sin declarar | Baja | `List<Integer>` explícito | Alta |
| Tipo de variables locales | Sin declarar | Baja | `Integer item`, `List<Integer> output` | Alta |
| Polimorfismo paramétrico | Ausente | N/A | `List<Integer>` con generics | Alta |
| Verificación del operador `%` | Duck typing en runtime | Baja | Nominal, en compilación | Alta |
| Coerción implícita | Sin coerción relevante | — | Autoboxing `Integer ↔ int` | Leve reducción |

### Conclusión

Ambas funciones son **semánticamente equivalentes** — calculan el mismo resultado — pero difieren radicalmente en cuándo y cómo el sistema de tipos garantiza su corrección:

- **Python** tiene el **grado de dureza más bajo**: ninguna de las características del sistema de tipos (tipo de parámetros, tipo de retorno, tipo de elementos de la lista, validez del operador `%`) es verificada antes de ejecutar el programa. Toda la responsabilidad recae en el programador y los errores se descubren en runtime.

- **Java** tiene el **grado de dureza más alto**: los tipos de parámetros, retorno, variables locales y elementos de la lista son declarados explícitamente y verificados en compilación. El sistema de generics garantiza la homogeneidad del tipo de elemento. La única concesión es el autoboxing implícito de `Integer ↔ int`, que introduce una muy leve blandura.
