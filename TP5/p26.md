## Pregunta 26
> ¿Qué formas de polimorfismo soporta Java, Pascal y Python?

---

## Respuesta

### Taxonomía de referencia

El sistema de tipos de un lenguaje define, entre otras cosas, su **nivel de polimorfismo**. La taxonomía usada en la materia distingue:

```
Polimorfismo
├── Ad-hoc (aparente: no hay una entidad ligada a varios tipos, sino entidades distintas con mismo nombre)
│   ├── Coerción
│   └── Sobrecarga
└── Universal (real: una misma entidad puede ligarse a tipos distintos)
    ├── Paramétrico
    └── Por inclusión
```

> *"Las opciones del llamado polimorfismo ad-hoc están dentro de la taxonomía, pero se trata de un nivel aparente de polimorfismo. Es aparente porque es una ilusión que se le vende al programador. Internamente esas entidades no son polimórficas. [...] Para entender por qué las operaciones sobrecargadas no son estrictamente polimórficas, la clave está en lo que definimos recién. No hay una operación que recibe operandos de distintos tipos, sino varias operaciones monomórficas que coinciden en el nombre."*
> — 06-SistemaDeTipos (p. 3)

> *"Overloaded subprograms provide a particular kind of polymorphism called ad hoc polymorphism. Overloaded subprograms need not behave similarly. Languages that support object-oriented programming usually support subtype polymorphism. [...] Parametric polymorphism is provided by a subprogram that takes generic parameters that are used in type expressions that describe the types of the parameters of the subprogram. [...] C++, Java 5.0+, C# 2005+, and F# provide a kind of compile-time parametric polymorphism."*
> — Sebesta, Concepts of Programming Languages (p. 656)

### 1. Pascal (estándar — ISO 7185)

Pascal estándar fue diseñado antes del paradigma orientado a objetos y tiene el nivel de polimorfismo más limitado de los tres lenguajes.

| Forma | Soporte |
|---|---|
| Coerción (ad-hoc) | ✅ Limitada: solo `integer → real` implícito |
| Sobrecarga (ad-hoc) | ⚠️ Parcial: solo operadores aritméticos built-in (`+`, `-`, `*` funcionan sobre `integer` y `real`); el usuario no puede definir sobrecarga |
| Por inclusión (universal) | ❌ No tiene clases ni herencia en el estándar |
| Paramétrico (universal) | ❌ No tiene genéricos |

Pascal estándar es **monomórfico** desde el punto de vista del programador: no puede definir funciones con el mismo nombre para distintos tipos, ni funciones genéricas, ni usar variables polimórficas por herencia.

> Nota: Object Pascal (Turbo Pascal, Delphi) extiende el estándar añadiendo clases, herencia simple y métodos virtuales, lo que incorpora **polimorfismo por inclusión**. Delphi también incorporó generics en versiones posteriores.

### 2. Java

Java ofrece todas las formas de polimorfismo de la taxonomía.

#### 2.1 Coerción (ad-hoc)

Java realiza conversiones numéricas implícitas de tipo expansor (*widening*): `byte → short → int → long → float → double`. Estas se aplican en expresiones mixtas, asignaciones y pasaje de parámetros.

```java
int i = 5;
double d = i;      // int → double implícito
double r = i + 3.14;  // i coercionado a double antes de sumar
```

#### 2.2 Sobrecarga (ad-hoc)

Java permite **sobrecarga de métodos**: el mismo nombre con distintas firmas (tipos o cantidad de parámetros). El compilador resuelve la versión correcta en tiempo de compilación.

```java
void imprimir(int x)    { System.out.println("int: " + x); }
void imprimir(String s) { System.out.println("str: " + s); }

imprimir(42);      // → int: 42
imprimir("hola");  // → str: hola
```

Java **no permite** sobrecarga de operadores definida por el usuario.

#### 2.3 Polimorfismo por inclusión (universal)

Una variable de tipo `T` puede referenciar objetos de cualquier subtipo de `T` (herencia simple de clases, implementación múltiple de interfaces). La resolución del método concreto se hace en tiempo de ejecución (*dynamic dispatch*).

```java
abstract class Animal { abstract void hablar(); }
class Perro extends Animal { void hablar() { System.out.println("Guau"); } }
class Gato  extends Animal { void hablar() { System.out.println("Miau"); } }

Animal a = new Perro();  // variable de tipo Animal, objeto Perro
a.hablar();              // → "Guau"  (binding dinámico)
a = new Gato();
a.hablar();              // → "Miau"
```

> *"Subtype polymorphism means that a variable of type T can access any object of type T or any type derived from T."*
> — Sebesta, Concepts of Programming Languages (p. 656)

Para garantizar la sustitución segura, la relación de herencia debe respetar la relación *is-a*:

> *"Para evitar las violaciones de tipos se debe poder sustituir a un objeto de una clase derivada por un objeto de la clase base en cualquier contexto. [...] La clase derivada solo puede extender la funcionalidad de la clase base."*
> — 06-SistemaDeTipos (p. 4)

#### 2.4 Polimorfismo paramétrico (universal)

Desde Java 5.0, el lenguaje ofrece **generics**. Una única definición de clase o método trabaja con cualquier tipo pasado como parámetro tipo.

```java
// Una sola clase genérica, instanciable para cualquier tipo T
class Caja<T> {
    private T valor;
    public Caja(T v) { this.valor = v; }
    public T obtener() { return valor; }
}

Caja<Integer> cajaInt = new Caja<>(42);
Caja<String>  cajaStr = new Caja<>("hola");
```

Implementado mediante **type erasure**: los parámetros de tipo son verificados en compilación y eliminados en el bytecode.

> *"C++, Java 5.0+, C# 2005+, and F# provide a kind of compile-time parametric polymorphism."*
> — Sebesta, Concepts of Programming Languages (p. 656)

### 3. Python

Python tiene un perfil de polimorfismo distinto al de Java: privilegia la flexibilidad en tiempo de ejecución sobre las verificaciones estáticas.

#### 3.1 Coerción (ad-hoc)

Solo para tipos numéricos: `integer → float → complex`. Fuera de ese contexto, Python es estricto.

```python
x = 1
y = 1.1
x + y      # ⇒ 2.1  (integer coercionado a float)
x + True   # ⇒ 2    (True coercionado a 1)
"hola" + 1 # ⇒ Error  (sin coerción entre string y número)
```

> *"Para tipos numéricos, realiza conversiones hacia tipos más generales. Un entero se convierte a real o un real a complejo."*
> — Slides - Sistema de tipos - Python y TypeScript (p. 15)

#### 3.2 Sobrecarga (ad-hoc)

Python **no permite sobrecarga de funciones** por nombre (si se redefine `f`, la primera definición queda descartada). Sí permite **sobrecarga de operadores** mediante *dunder methods* (`__add__`, `__mul__`, etc.), tanto para operadores built-in como para clases definidas por el usuario.

```python
def f1(x):     print(x)
def f1(x, y):  print(x)  # reemplaza a la anterior

f1(4)     # ⇒ Error (no existe f1 de un parámetro)
```

```python
class CustomList:
    def __init__(self, elems): self.elements = elems
    def __add__(self, other):
        return CustomList(self.elements + [other])
lista = CustomList([1, 2]) + 3  # ⇒ [1, 2, 3]
```

> *"Python no permite sobrecarga de funciones, pero sí cuenta con operadores sobrecargados."*
> — Slides - Sistema de tipos - Python y TypeScript (p. 18)

#### 3.3 Polimorfismo por inclusión (universal)

Python soporta herencia simple y **múltiple**. Los métodos son resueltos dinámicamente. Sin embargo, Python no garantiza el principio de sustitución: una clase derivada puede redefinir métodos con comportamiento incompatible con la clase base.

```python
class BaseClass:
    def get_message(self): return (self.v1, self.v2)

class Derivada(BaseClass):
    def get_message(self): return 123  # retorna int, no tupla

len(Derivada().get_message())  # ⇒ Error: int no tiene len
```

> *"Una clase derivada puede definir comportamiento que no es compatible con el definido en su clase ancestro."*
> — Slides - Sistema de tipos - Python y TypeScript (p. 21)

> *"Python brinda herencia simple y múltiple."*
> — Slides - Sistema de tipos - Python y TypeScript (p. 20)

#### 3.4 Polimorfismo paramétrico / Duck Typing (universal)

Python **no soporta polimorfismo paramétrico** en el sentido estático (no hay generics en tiempo de ejecución). En cambio, usa **duck typing**: las variables no tienen tipo declarado, y una función acepta cualquier objeto que tenga los atributos o métodos que usa, independientemente de su tipo nominal. Esto da un nivel de flexibilidad aún mayor que el polimorfismo paramétrico.

```python
class Pato:    def hablar(self): print("Quack!")
class Persona: def hablar(self): print("Hola")

def hacer_hablar(x):
    x.hablar()  # funciona para cualquier objeto con método hablar()

hacer_hablar(Pato())    # ⇒ "Quack!"
hacer_hablar(Persona()) # ⇒ "Hola"
```

> *"Python no soporta polimorfismo paramétrico."*
> — Slides - Sistema de tipos - Python y TypeScript (p. 20)

> *"Python utiliza el concepto de duck typing para realizar parte de los chequeos de tipos. Al usar duck typing, los atributos y métodos de un objeto resultan más relevantes que el tipo del objeto."*
> — Slides - Sistema de tipos - Python y TypeScript (p. 22)

> *"A more general kind of polymorphism is provided by the methods of Python and Ruby. Recall that variables in these languages do not have types, so formal parameters do not have types. Therefore, a method will work for any type of actual parameter, as long as the operators used on the formal parameters in the method are defined."*
> — Sebesta, Concepts of Programming Languages (p. 656)

### Resumen comparativo

| Forma de polimorfismo | Pascal estándar | Java | Python |
|---|---|---|---|
| **Coerción** (ad-hoc) | ✅ Solo `integer → real` | ✅ Widening numérico (`byte → ... → double`) | ✅ Solo numérico (`int → float → complex`) |
| **Sobrecarga** (ad-hoc) | ⚠️ Solo operadores built-in; sin sobrecarga de usuario | ✅ Métodos; sin sobrecarga de operadores de usuario | ⚠️ Operadores vía dunder; sin sobrecarga de funciones |
| **Por inclusión** (universal) | ❌ (✅ en Object Pascal) | ✅ Herencia simple + interfaces | ✅ Herencia simple y múltiple |
| **Paramétrico** (universal) | ❌ | ✅ Generics desde Java 5.0 | ❌ (reemplazado por duck typing) |
| **Duck typing** | ❌ | ❌ | ✅ Característica central del lenguaje |
