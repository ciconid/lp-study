## Pregunta 3
> Indique cuáles son los principales mecanismos que se pueden utilizar para definir la ejecución concurrente de un bloque de código. Explique brevemente el funcionamiento de cada uno de estos mecanismos y analice sus ventajas y desventajas.

---

## Respuesta

Desde el punto de vista del diseño de lenguajes, existen distintas categorías de tareas que pueden usarse para concurrencia:

> *"Existen dos categorías a considerar: Heavyweight task: Son tareas que cuentan con su propio espacio de direcciones y su propia pila de ejecución. Lightweight task: En este caso todas las tareas se ejecutan sobre el mismo espacio de nombre y comparten la pila de ejecución."*
> — 09-Concurrencia.pdf (p. 2)

Los principales mecanismos para **definir** la ejecución concurrente son:

---

### 1. Fork / Join

El mecanismo fork/join (originalmente de PL/I) consiste en dos operaciones:
- **Fork**: crea una nueva tarea que ejecuta un subprograma determinado; el llamador continúa su ejecución sin esperar.
- **Join**: el llamador se suspende hasta que la tarea forkeada complete su ejecución.

```
tid = fork(subprograma, args)  -- lanza concurrentemente
...                             -- continúa en paralelo
join(tid)                       -- espera que termine
```

**Ventajas:**
- Simple y flexible: cualquier función puede ejecutarse concurrentemente.
- No requiere constructores especiales del lenguaje más allá de dos primitivas.

**Desventajas:**
- Mecanismo no estructurado: puede olvidarse hacer el `join`, generando tareas huérfanas.
- Propenso a errores: el programador debe gestionar manualmente los identificadores de tareas y su ciclo de vida.
- Sin garantías de completitud en caso de error.

---

### 2. Cobegin / Coend (bloques paralelos)

Un bloque `cobegin/coend` (también llamado `par/begin` o `parbegin/parend` en algunas notaciones) especifica un conjunto de sentencias o llamadas que deben ejecutarse en paralelo. El bloque completa cuando **todas** las ramas han finalizado.

```
cobegin
  tarea1();
  tarea2();
  tarea3();
coend
```

**Ventajas:**
- Estructurado: queda claro en el código qué unidades se ejecutan en paralelo y dónde se sincronizan.
- El bloque tiene una semántica clara de punto de entrada y punto de salida.
- Más fácil de razonar y de verificar.

**Desventajas:**
- Inflexible: el número y tipo de tareas a ejecutar deben conocerse en el momento de ingresar al bloque.
- No escala bien para concurrencia dinámica (e.g., crear tareas en función de datos de entrada en tiempo de ejecución).

---

### 3. Declaración de tareas (Ada)

Ada provee un mecanismo declarativo para definir tareas. Una tarea tiene dos partes sintácticas: una **especificación** (que define su interfaz de entrada) y un **cuerpo** (que define su comportamiento).

> *"There are two syntactic parts to an Ada task—a specification part and a body part—both with the same name."*
> — Concepts of Programming Languages - Sebesta - E12.pdf (p. 890)

Las tareas Ada son **heavyweight**: ejecutan en su propio espacio de direcciones.

> *"Java's threads are lightweight tasks, which means that they all run in the same address space. This is different from Ada tasks, which are heavyweight threads (they run in their own address spaces)."*
> — Concepts of Programming Languages - Sebesta - E12.pdf (p. 903)

Una tarea Ada se activa implícitamente cuando el bloque que la contiene es elaborado, sin necesidad de llamarla explícitamente.

```ada
task Task_Example is
  entry Entry_1(Item : in Integer);
end Task_Example;

task body Task_Example is
begin
  loop
    accept Entry_1(Item : in Integer) do
      -- código del rendezvous
    end Entry_1;
  end loop;
end Task_Example;
```

**Ventajas:**
- Integrado en el lenguaje: el compilador puede verificar el uso correcto de tareas.
- El inicio implícito reduce boilerplate.
- La interfaz declarativa (specification) separa claramente interfaz de implementación.

**Desventajas:**
- Mayor overhead: las tareas heavyweight consumen más recursos que los hilos ligeros.
- El inicio implícito puede dificultar la comprensión del flujo de control del programa.

---

### 4. Clase Thread / interfaz Runnable (Java, C#)

En lenguajes orientados a objetos, la concurrencia se define creando una subclase de `Thread` (o implementando `Runnable`) y sobreescribiendo el método `run()`. La tarea se lanza llamando a `start()`.

> *"The run method, which describes the actions of the thread, is always overridden by subclasses of Thread. The start method of Thread starts its thread as a concurrent unit by calling its run method. The call to start is unusual in that control returns immediately to the caller, which then continues its execution, in parallel with the newly started run method."*
> — Concepts of Programming Languages - Sebesta - E12.pdf (p. 903)

Los hilos Java son **lightweight**: comparten el mismo espacio de direcciones.

```java
class MyThread extends Thread {
    public void run() { /* código concurrente */ }
}
Thread t = new MyThread();
t.start();  // lanza la tarea; el llamador continúa
```

**Ventajas:**
- Integración natural con el paradigma OOP.
- Hilos ligeros: bajo overhead de creación y gestión.
- La clase `Thread` provee métodos útiles (`join()`, `sleep()`, `interrupt()`, etc.).

**Desventajas:**
- Más verboso que mecanismos tipo `go` o `fork`.
- La herencia única de Java obliga a usar `Runnable` cuando la clase ya tiene un padre natural.
- La sincronización queda a cargo del programador (con `synchronized`, `wait()`, `notify()`).

---

### 5. Sentencia `go` — Goroutines (Go)

Go provee la sentencia `go` que lanza cualquier llamada de función como una **goroutine**: una abstracción ligera de hilo administrada por el runtime de Go.

> *"A diferencia de otros lenguajes de programación que utilizan hilos del sistema operativo, Go permite ejecutar tareas de forma concurrente utilizando goroutines. Una goroutine es una abstracción de un hilo del sistema operativo. Un planificador en el runtime de Go se encargará de asignar goroutines a hilos del sistema operativo para que sean ejecutadas."*
> — Slides - Concurrencia y Go.pdf (p. 31)

```go
go saludar()           // lanza saludar() como goroutine
go func() { ... }()    // goroutine con función anónima
```

**Ventajas:**
- Sintaxis mínima: una sola palabra clave lanza una tarea.
- Muy bajo overhead: el runtime multiplexa miles de goroutines sobre pocos hilos del SO.
- No requiere herencia ni clases especiales.

**Desventajas:**
- Sin mecanismo built-in de espera al finalizar: se necesita un `sync.WaitGroup` o un channel para sincronizar.
- El programador debe gestionar explícitamente la coordinación y el ciclo de vida de las goroutines.

---

### 6. Concurrencia a nivel sentencia — FORALL (High-Performance Fortran)

Para concurrencia a nivel de sentencia, High-Performance Fortran introduce la construcción `FORALL`, que indica al compilador que las iteraciones de un loop pueden ejecutarse en paralelo.

> *"The FORALL statement specifies a sequence of assignment statements that may be executed concurrently."*
> — Concepts of Programming Languages - Sebesta - E12.pdf (p. 928)

```fortran
FORALL (index = 1:1000)
  list_1(index) = list_2(index)
END FORALL
```

**Ventajas:**
- Permite aprovechar arquitecturas vectoriales/SIMD sin cambiar la lógica del programa.
- Simple de expresar para ciertos patrones regulares (loops sobre arrays).

**Desventajas:**
- Limitado a patrones muy específicos: las iteraciones deben ser independientes.
- El programador debe garantizar que no hay dependencias entre iteraciones; el compilador no siempre puede verificarlo.
- No es aplicable a concurrencia general (lógica de negocio, I/O concurrente, etc.).

---

### Resumen comparativo

| Mecanismo | Granularidad | Estructurado | Overhead | Inicio |
|---|---|---|---|---|
| Fork/Join | Subprograma | No | Medio | Explícito |
| Cobegin/Coend | Bloque | Sí | Medio | Implícito |
| Task Ada | Unidad | Sí | Alto (heavyweight) | Implícito |
| Thread Java/C# | Objeto/método | Semi | Bajo (lightweight) | Explícito |
| Goroutine Go | Función | No | Muy bajo | Explícito (`go`) |
| FORALL (HPF) | Sentencia | Sí | Bajo | Implícito (compilador) |
