## Pregunta 22
> Considere las siguientes sentencias en Java:
> - a. `double v1 = 4.0 / (5 / 10);`
> - b. `double v2 = 4.0 / (5.0 / 10);`
>
> ¿Qué diferencias ve entre a) y b)? ¿Qué valores quedarán almacenados en v1 y v2 respectivamente luego de ejecutar esas sentencias? ¿Por qué son diferentes?

---

## Respuesta

### 1. Contexto: expresiones mixtas y coerción en Java

Java permite **expresiones mixtas** — aquellas en que los operandos son de tipos distintos — y las resuelve insertando coerciones implícitas de ensanchamiento (*widening*) donde corresponde.

> *"Languages that allow mixed-mode expressions must define conventions for implicit operand type conversions because computers do not have binary operations that take operands of different types. […] Because mixed-mode expressions are legal in Java, the compiler would not detect this as an error. It would simply insert code to coerce the value of the int operand to float."*
> — Sebesta, Concepts of Programming Languages (p. 522–523)

Las coerciones en Java son únicamente expansoras: un tipo más pequeño puede convertirse implícitamente en uno más grande (`int` → `double`), pero no al revés.

> *"Java includes assignment type coercions (implicit type conversions) only if they are widening (from a 'smaller' type to a 'larger' type). So int to float coercions are done across the assignment operator, but float to int coercions are not."*
> — Sebesta, Concepts of Programming Languages (p. 175)

> *"Las conversiones expansoras se consideran seguras, precisamente porque no generan pérdida de información, solamente generan una modificación en la forma en la que un determinado valor está almacenado."*
> — 06-SistemaDeTipos (p. 2)

Un detalle crucial es que Java evalúa las subexpresiones **de adentro hacia afuera**, aplicando las reglas de tipo al nivel exacto en que los operandos se encuentran — **no** en función del tipo de la variable destino de la asignación.

---

### 2. Caso a: `double v1 = 4.0 / (5 / 10)`

**Paso 1 — Evaluar `(5 / 10)`:**

Los literales `5` y `10` son ambos de tipo `int`. Al ser los dos operandos enteros, Java aplica la **división entera** (división truncada hacia cero):

$$5 \div 10 = 0.5 \xrightarrow{\text{truncación}} 0 \quad (\text{tipo: int})$$

El hecho de que el resultado vaya a usarse en un contexto `double` no influye en la evaluación de esta subexpresión.

**Paso 2 — Evaluar `4.0 / 0`:**

Ahora los operandos son `4.0` (`double`) y `0` (`int`): expresión mixta. Java widena `0` → `0.0` (`double`).

La expresión resultante es `4.0 / 0.0`. En Java, dividir un `double` positivo por `0.0` **no lanza una excepción**: produce el valor especial `Double.POSITIVE_INFINITY` (definido por IEEE 754).

**Paso 3 — Asignación:**

```java
double v1 = Double.POSITIVE_INFINITY;
```

**Resultado: `v1 = Infinity`**

---

### 3. Caso b: `double v2 = 4.0 / (5.0 / 10)`

**Paso 1 — Evaluar `(5.0 / 10)`:**

Los operandos son `5.0` (`double`) y `10` (`int`): expresión mixta. Java widena `10` → `10.0` (`double`).

La expresión resultante es `5.0 / 10.0`, división en punto flotante:

$$5.0 \div 10.0 = 0.5 \quad (\text{tipo: double})$$

**Paso 2 — Evaluar `4.0 / 0.5`:**

Ambos operandos son `double`, no hay coerción. División normal:

$$4.0 \div 0.5 = 8.0$$

**Paso 3 — Asignación:**

```java
double v2 = 8.0;
```

**Resultado: `v2 = 8.0`**

---

### 4. ¿Por qué son diferentes?

La diferencia radica exclusivamente en el tipo de los operandos de la **subexpresión interna**:

| Subexpresión  | Tipos de operandos | Tipo de división        | Resultado |
|---------------|--------------------|-------------------------|-----------|
| `5 / 10`      | `int` / `int`      | Entera (con truncación) | `0`       |
| `5.0 / 10`    | `double` / `int`   | Flotante (con widening) | `0.5`     |

Cambiar `5` por `5.0` hace que el operando sea `double`, lo que fuerza a Java a aplicar una coerción expansora sobre el `10` y a resolver la división en punto flotante, preservando la parte fraccional.

El tipo de la variable destino (`double v1`, `double v2`) **no retropropaga** su tipo hacia las subexpresiones internas: Java no es especulativo en sus coerciones. Cada operador resuelve su tipo según los operandos que tiene en ese momento.

> *"In a clear departure from C++, Java and C# allow mixed-mode assignment only if the required coercion is widening. So, an int value can be assigned to a float variable, but not vice versa."*
> — Sebesta, Concepts of Programming Languages (p. 539)

Esto ilustra además una limitación de la detección de errores en Java: al permitir expresiones mixtas, errores como el de `a)` no son señalados por el compilador — el programador recibe `Infinity` sin ningún aviso, lo que puede provocar bugs silenciosos.

> *"Because error detection is reduced when mixed-mode expressions are allowed, F#, Ada, and ML do not allow them."*
> — Sebesta, Concepts of Programming Languages (p. 523)

---

### Resumen

| Variable | Expresión               | Valor final  | Razón                                              |
|----------|-------------------------|--------------|----------------------------------------------------|
| `v1`     | `4.0 / (5 / 10)`        | `Infinity`   | `5/10` es división entera → `0`; `4.0/0.0` = ∞    |
| `v2`     | `4.0 / (5.0 / 10)`      | `8.0`        | `5.0/10` es flotante → `0.5`; `4.0/0.5` = `8.0`  |
