## Respuesta

Go tiene tipado estático y provee cuatro tipos estructurados fundamentales: arrays, slices, structs y maps.

> *"Como tipos estructurados, se destacan los arrays, slices, structs y maps."*
> — Slides - Concurrencia y Go (p. 21)

---

### 1. Arrays

#### Decisión de diseño: el tamaño es parte del tipo

En Go, el tamaño de un array forma parte de su tipo. `[5]int` y `[10]int` son **tipos distintos** e incompatibles. Los arrays tienen **semántica de valor**: asignar o pasar un array crea una copia completa de todos sus elementos.

```go
var a [2]int          // tipo: [2]int
b := [5]int{1,2,3,4,5} // tipo: [5]int
// a = b  → error de compilación: [2]int != [5]int

func foo(arr [4]int) { arr[0] = 0 }
x := [4]int{1, 2, 3, 4}
foo(x)
fmt.Println(x[0])  // ⇒ 1  (la copia no afecta al original)
```

> *"Para tipos básicos, arrays, structs y punteros, se crea una copia del valor pasado como argumento al parámetro formal de la función invocada."*
> — Slides - Concurrencia y Go (p. 29)

#### Contraste: Java

En Java, los arrays son objetos de tipo de referencia: el tamaño no es parte del tipo estático (`int[]` es el tipo de cualquier arreglo de enteros, independientemente de su tamaño). Asignar o pasar un array copia solo la referencia, no los elementos.

```java
int[] a = new int[2];
int[] b = new int[5];
// a y b son del mismo tipo: int[]

int[] c = a;  // c y a apuntan al mismo arreglo
c[0] = 99;
System.out.println(a[0]);  // ⇒ 99  (se modificó el original)
```

> *"In Java, all non-generic arrays are fixed heap-dynamic. Once created, these arrays keep the same subscript ranges and storage."*
> — Sebesta, Concepts of Programming Languages (p. 425)

**Impacto**: en Go, incluir el tamaño en el tipo permite al compilador detectar usos incorrectos de arreglos de tamaño distinto. En Java, se pierde esa verificación y la semántica de referencia puede generar efectos colaterales inadvertidos.

### 2. Slices

#### Decisión de diseño: longitud y capacidad separadas; vista sobre un array subyacente

Un slice en Go es un **descriptor** compuesto de tres campos: un puntero al array subyacente, una longitud (`len`) y una capacidad (`cap`). Múltiples slices pueden compartir el mismo array subyacente. El slice no posee el almacenamiento, solo lo referencia.

```go
s := make([]int, 3)
fmt.Println("length:", len(s))   // ⇒ 3
fmt.Println("capacity:", cap(s)) // ⇒ 3

s = append(s, 1)
fmt.Println("slice:", s)         // ⇒ [0 0 0 1]
fmt.Println("length:", len(s))   // ⇒ 4

// Dos slices sobre el mismo array
arr := [5]int{1, 2, 3, 4, 5}
s1 := arr[0:3]  // [1 2 3], cap=5
s2 := arr[1:4]  // [2 3 4], cap=4
s1[1] = 99
fmt.Println(s2[0])  // ⇒ 99  (comparten el array)
```

> *"Para strings, slices, maps o channels, se crea una copia del descriptor que contiene punteros a los datos en memoria."*
> — Slides - Concurrencia y Go (p. 30)

#### Contraste: Python (`list`)

Python listas crecen y se reducen dinámicamente pero **no exponen la capacidad** al programador ni tienen la noción de "vista sobre un array compartido". Cada lista es completamente independiente; las modificaciones a una copia no afectan a la original salvo que se opere sobre el mismo objeto (por referencia).

```python
a = [1, 2, 3, 4, 5]
b = a[0:3]  # nueva lista, copia independiente
b[1] = 99
print(a[1])  # ⇒ 2  (no se modificó a)
```

**Impacto**: en Go, la noción de capacidad permite controlar las realocaciones del array subyacente y es relevante para el rendimiento. La vista compartida es poderosa pero requiere cuidado para evitar efectos colaterales. Python es más simple pero menos eficiente para operaciones de bajo nivel sobre memoria.

### 3. Structs

#### Decisión de diseño: semántica de valor; sin herencia (composición por embedding)

Los structs en Go tienen **semántica de valor por defecto**: se copian al asignarse o pasarse como argumento. Go no tiene herencia de tipos; la reutilización de comportamiento se logra mediante **embedding** (composición estructural).

```go
type Person struct {
    Name string
    Age  int
}

type Employee struct {
    Person        // embedding: Employee "tiene un" Person
    Company string
}

func main() {
    e := Employee{Person: Person{"Ana", 30}, Company: "UNS"}
    fmt.Println(e.Name)    // ⇒ "Ana"  (campos promovidos)
    fmt.Println(e.Age)     // ⇒ 30

    // Semántica de valor: copia
    e2 := e
    e2.Name = "Luis"
    fmt.Println(e.Name)    // ⇒ "Ana"  (el original no cambia)
}
```

#### Contraste: Java (clases con semántica de referencia y herencia)

En Java, los objetos son siempre **tipos de referencia**: una variable contiene una referencia, no el valor. La reutilización de comportamiento se hace principalmente mediante **herencia** (`extends`), no composición.

```java
class Person {
    String name; int age;
}
class Employee extends Person {  // herencia
    String company;
}

Employee e1 = new Employee();
Employee e2 = e1;   // e2 apunta al mismo objeto
e2.name = "Luis";
System.out.println(e1.name);  // ⇒ "Luis"  (misma referencia)
```

**Impacto**: la semántica de valor de Go reduce el riesgo de efectos colaterales inesperados pero puede ser costosa para structs grandes (se copian todos los campos). Java favorece la mutabilidad compartida. La ausencia de herencia en Go obliga a preferir composición, lo que en general se considera más mantenible pero requiere un cambio de mentalidad para programadores OO.

### 4. Maps

#### Decisión de diseño: valor cero `nil`; claves deben ser comparables; orden de iteración no determinista

En Go, el valor cero de un map es `nil`. Un map `nil` **no puede recibir escrituras** sin causar `panic`; debe inicializarse explícitamente con `make`. Las claves deben ser de tipos **comparables** (sin slices ni maps como claves). La iteración sobre un map es **intencionalmente no determinista**: el lenguaje deliberadamente aleatoriza el orden para evitar que el programador dependa de él.

```go
m := make(map[string]int)
m["k1"] = 7
m["k2"] = 13
fmt.Println("map:", m)  // map: map[k1:7 k2:13]

delete(m, "k2")
v1, ok := m["k1"]
fmt.Println("v1:", v1, "ok:", ok)  // v1: 7 ok: true

v2, ok := m["inexistente"]
fmt.Println("v2:", v2, "ok:", ok)  // v2: 0 ok: false

// var m2 map[string]int  →  m2 == nil
// m2["x"] = 1  →  panic: assignment to entry in nil map
```

> *"Un map permite definir una colección de pares clave-valor. Para declarar un map usaremos la función make. Al definirlo, debemos indicar el tipo para las claves y para los valores."*
> — Slides - Concurrencia y Go (p. 22)

#### Contraste: Python (`dict`)

Python preserva el **orden de inserción** de las claves desde Python 3.7 (garantizado por la especificación). Además, cualquier objeto **hashable** puede ser clave (incluyendo tuplas), y el valor cero del diccionario es un dict vacío `{}`, no `nil` — está listo para usar sin inicialización explícita.

```python
d = {}           # listo para usar, no es nil
d["k1"] = 7
d["k2"] = 13
for k, v in d.items():
    print(k, v)  # garantiza el orden de inserción: k1, k2

d[(1, 2)] = "tupla como clave"  # válido en Python
```

**Impacto**: el no-determinismo del orden en Go es una decisión deliberada para evitar dependencia en el orden de iteración (que podría variar entre implementaciones). Python, al preservar el orden, facilita ciertos patrones pero puede dar la impresión de que el orden siempre importa. La restricción de claves comparables en Go brinda mayor seguridad de tipos; Python usa `hash()` y acepta más tipos.

> *"Perl includes associative arrays, which are called hashes. These data structures are indexed by strings and are implicitly controlled hash tables."*
> — Sebesta, Concepts of Programming Languages (p. 179)

### Resumen comparativo

| Tipo    | Decisión de diseño en Go                                   | Lenguaje de contraste  | Decisión diferente                                                  |
|---------|------------------------------------------------------------|------------------------|----------------------------------------------------------------------|
| Array   | Tamaño parte del tipo; semántica de valor                  | Java                   | Tamaño no es parte del tipo; semántica de referencia                |
| Slice   | Len y cap separados; vista sobre array subyacente          | Python `list`          | Sin capacidad expuesta; copia independiente al hacer slice          |
| Struct  | Semántica de valor; composición por embedding (sin herencia) | Java (clases)         | Semántica de referencia; reutilización por herencia                 |
| Map     | Valor cero nil; claves comparables; iteración no determinista | Python `dict`        | Vacío listo para usar; orden de inserción garantizado; más tipos de clave |
