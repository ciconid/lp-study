## Pregunta 32

Analice el siguiente código Python:

```python
def calcular_descuento(precio: float, porcentaje: int) -> float:
    return precio * (1 - porcentaje / 100)

def aplicar_descuentos(precios: list[float], descuento: int) -> list[float]:
    return [calcular_descuento(p, descuento) for p in precios]
```

Investigue en qué consisten los type hints (o type annotations) en Python y cómo impactan en el chequeo de tipos del lenguaje. ¿Cambia el comportamiento del intérprete? ¿Qué ventajas brindan para el desarrollo? Para complementar su respuesta, puede explorar la herramienta mypy.

---

## Respuesta

### Python como lenguaje de tipado dinámico (base)

Antes de analizar el código, hay que recordar el punto de partida:

> *"Es un lenguaje interpretado con tipado dinámico, fuertemente tipado. Los chequeos de tipos se realizan en tiempo de ejecución."*
> — Slides - Sistema de tipos - Python y TypeScript (p. 5)

En Python, las variables no se declaran con un tipo; el tipo se liga al **valor** en tiempo de ejecución, no a la variable. Esto hace que el chequeo estático sea, por defecto, imposible: el intérprete no tiene información de tipo en compilación.

---

### ¿Qué son los type hints?

Los **type hints** (o *type annotations*) son anotaciones sintácticas introducidas en Python 3.5 (PEP 484) que permiten asociar información de tipo a variables, parámetros y retornos de funciones. En el código del enunciado:

```python
def calcular_descuento(precio: float, porcentaje: int) -> float:
```

- `precio: float` → el parámetro `precio` se espera que sea de tipo `float`
- `porcentaje: int` → el parámetro `porcentaje` se espera que sea `int`
- `-> float` → la función retorna un `float`

```python
def aplicar_descuentos(precios: list[float], descuento: int) -> list[float]:
```

- `list[float]` (Python 3.9+) → una lista cuyos elementos se esperan `float`

Las anotaciones pueden también aplicarse a variables locales:
```python
x: int = 5
nombre: str
```

---

### ¿Cambia el comportamiento del intérprete?

**No.** El intérprete de Python **ignora completamente las type annotations en tiempo de ejecución**. Son puro metadata.

```python
def calcular_descuento(precio: float, porcentaje: int) -> float:
    return precio * (1 - porcentaje / 100)

# El intérprete NO verifica los tipos:
calcular_descuento("cien", "veinte")   # no hay error al llamar
# Solo fallará cuando intente ejecutar "cien" * (1 - "veinte" / 100)
# → TypeError: unsupported operand type(s) for /: 'str' and 'int'
```

La única huella en tiempo de ejecución es que las anotaciones quedan almacenadas en el atributo `__annotations__` del objeto función, accesible como metadata:

```python
>>> calcular_descuento.__annotations__
{'precio': <class 'float'>, 'porcentaje': <class 'int'>, 'return': <class 'float'>}
```

Esto confirma que las anotaciones son **documentación estructurada**, no restricciones de runtime. El chequeo de tipos sigue siendo dinámico, exactamente igual que sin anotaciones.

> *"Ante un error de tipos, Python lanzará una excepción correspondiente."*
> — Slides - Sistema de tipos - Python y TypeScript (p. 24)

Esa excepción seguirá ocurriendo en runtime con o sin type hints.

---

### Tipado gradual: mypy como herramienta de análisis estático

Lo que las type hints *sí* habilitan es el uso de herramientas de **análisis estático externas** que leen las anotaciones y reportan errores *antes de ejecutar el programa*. La más popular es **mypy**.

#### ¿Qué hace mypy?

mypy es un verificador de tipos estático para Python. Analiza el código fuente, lee las anotaciones y emite errores cuando los tipos no coinciden, **sin ejecutar el programa**.

```python
# archivo descuentos.py
def calcular_descuento(precio: float, porcentaje: int) -> float:
    return precio * (1 - porcentaje / 100)

calcular_descuento("cien", "veinte")   # error de tipos
```

```bash
$ mypy descuentos.py
descuentos.py:4: error: Argument 1 to "calcular_descuento" has incompatible type "str"; expected "float"
descuentos.py:4: error: Argument 2 to "calcular_descuento" has incompatible type "str"; expected "int"
```

mypy detecta el error **en tiempo de análisis estático**, antes de que el programa se ejecute.

#### Tipado gradual (*gradual typing*)

Una característica central de este enfoque es que es **gradual**: se puede tener código con anotaciones y código sin anotaciones en el mismo proyecto. mypy trata el código sin anotar como si fuera de tipo `Any` (compatible con todo), y solo reporta errores en las partes que tienen anotaciones.

Esto permite:
- Adoptar type hints incrementalmente en proyectos existentes
- Anotar solo las partes críticas del código
- No romper la naturaleza dinámica del lenguaje

> *"Tipo y tiempo de chequeo: el chequeo de tipo consiste en verificar que el tipo de una entidad corresponda al esperado por el contexto."*
> — 06-SistemaDeTipos (p. 1)

Con mypy, Python puede ahora realizar ese chequeo **estáticamente**, aunque de forma optativa.

---

### Análisis del código del enunciado con mypy

```python
def calcular_descuento(precio: float, porcentaje: int) -> float:
    return precio * (1 - porcentaje / 100)
```

mypy verifica:
- Que `precio * (...)` sea válido para `float` ✅
- Que `porcentaje / 100` sea válido para `int` ✅ (Python devuelve `float` para `/`)
- Que el resultado de la multiplicación sea `float` ✅

```python
def aplicar_descuentos(precios: list[float], descuento: int) -> list[float]:
    return [calcular_descuento(p, descuento) for p in precios]
```

mypy verifica:
- Que `p` (elemento de `list[float]`) sea compatible con el parámetro `precio: float` ✅
- Que `descuento: int` sea compatible con `porcentaje: int` ✅
- Que la comprensión devuelva `list[float]` ✅

Ambas funciones son **correctas para mypy** tal como están escritas.

---

### Ventajas de los type hints para el desarrollo

| Ventaja | Descripción |
|---|---|
| **Detección temprana de errores** | mypy detecta errores de tipos antes de ejecutar el programa, eliminando una clase de bugs que solo aparecerían en runtime |
| **Documentación embebida** | Los tipos son parte de la firma de la función, no de comentarios externos que pueden desactualizarse |
| **Soporte mejorado de IDEs** | VS Code, PyCharm y otros usan las anotaciones para autocompletado, detección de errores en el editor, y refactoring seguro |
| **Contratos más explícitos** | Las funciones expresan sus expectativas de tipos directamente, facilitando el trabajo en equipo y el mantenimiento |
| **Refactoring seguro** | Al cambiar una firma, mypy puede detectar todos los usos que se vuelven incompatibles |

---

### Comparación: Python sin y con type hints

| Aspecto | Python sin anotaciones | Python con type hints + mypy |
|---|---|---|
| Tiempo de chequeo de tipos | Solo runtime | Análisis estático + runtime |
| Obligatoriedad | — | Optativa (gradual) |
| Comportamiento del intérprete | Igual | Igual (ignoradas en runtime) |
| Errores detectados | Al ejecutar la línea con error | Antes de ejecutar (con mypy) |
| Flexibilidad | Máxima (duck typing) | Se mantiene; mypy es externo |
| Documentación de tipos | Implícita o en docstrings | Embebida en la firma |

> *"Python utiliza el concepto de duck typing para realizar parte de los chequeos de tipos. Al usar duck typing, los atributos y métodos de un objeto resultan más relevantes que el tipo del objeto."*
> — Slides - Sistema de tipos - Python y TypeScript (p. 22)

Los type hints **no eliminan el duck typing**: una función anotada con `precio: float` sigue aceptando cualquier objeto en runtime; mypy simplemente reportará un error si el argumento no satisface el tipo declarado en el análisis estático.
