## Pregunta 28
> Compare el grado de dureza del sistema de tipos de Python, Java y Pascal respecto a la sobrecarga de métodos.

---

## Respuesta

### Marco conceptual: dureza respecto a la sobrecarga

> *"La dureza del tipado no es una propiedad de los lenguajes, sino que es un grado que surge como consecuencia de las restricciones que impone el sistema de tipos."*
> — 06-SistemaDeTipos (p. 2)

Para la sobrecarga de métodos en particular, la dureza se evalúa por:

1. **Si el lenguaje permite sobrecarga definida por el usuario** (no permitirla es el extremo más restrictivo).
2. **Cómo resuelve la ambigüedad** cuando hay múltiples variantes con el mismo nombre: en tiempo de compilación (más duro) o en tiempo de ejecución (más blando).
3. **Si aplica coerciones implícitas** para encontrar una variante que coincida (hacerlo reduce la dureza).

> *"La sobrecarga necesita criterios definidos en el sistema de tipos para resolverse. [...] La incorporación de sobrecarga al lenguaje puede traer inconvenientes si se combina con otras características como el tipo unión, la utilización de parámetros por omisión y las expresiones mixtas."*
> — 06-SistemaDeTipos (p. 4)

---

### Pascal (estándar — ISO 7185)

Pascal estándar **no permite** sobrecarga de funciones o procedimientos definida por el usuario. Si el programador necesita la misma operación para distintos tipos, debe declarar subprogramas con nombres distintos.

```pascal
function MaxEntero(a, b: integer): integer;
function MaxReal(a, b: real): real;
{ No es posible tener dos funciones llamadas Max }
```

Las únicas "sobrecargas" existentes son las de los operadores aritméticos built-in (`+`, `-`, `*`, `/`), que el lenguaje maneja internamente convirtiendo entero a real cuando es necesario (coerción implícita expansora). Esta coerción es del lenguaje, no del programador.

**Dureza respecto a la sobrecarga**: **máxima** — el lenguaje directamente no permite definir sobrecarga de usuario, eliminando toda ambigüedad posible por la vía de la restricción. El programador no puede crear conflictos de sobrecarga porque la característica no existe.

---

### Java

Java **sí permite** sobrecarga de métodos. La resolución se hace **en tiempo de compilación** basándose en los tipos declarados de los argumentos y la cantidad de parámetros. No se permite sobrecarga solo por tipo de retorno.

```java
void imprimir(int x)    { System.out.println("int: " + x); }
void imprimir(String s) { System.out.println("str: " + s); }
void imprimir(double d) { System.out.println("double: " + d); }

imprimir(42);       // → int: 42     (resuelto en compilación)
imprimir("hola");   // → str: hola   (resuelto en compilación)
imprimir(3.14);     // → double: 3.14
```

Sin embargo, Java aplica **coerciones de widening** (ampliación de tipo) durante la resolución de sobrecarga cuando no existe una variante con coincidencia exacta:

```java
void foo(long x) { System.out.println("long"); }

int n = 5;
foo(n);  // int es ampliado a long → "long"
         // (no hay foo(int), Java aplica widening automático)
```

El compilador también sigue la regla de **método más específico**: si hay múltiples variantes que podrían aplicar, elige la más restrictiva. Si hay ambigüedad no resoluble, reporta error en compilación.

Java **no permite** sobrecarga de operadores definida por el usuario (solo existe para `String +`, que es built-in).

**Dureza respecto a la sobrecarga**: **alta, con ligera reducción por coerciones de widening**. La resolución es estática y temprana; los errores de ambigüedad se detectan en compilación. La aplicación de coerciones implícitas en la resolución introduce un pequeño grado de blandura, ya que el compilador puede elegir una variante que no coincida exactamente con los tipos provistos.

### Python

Python **no permite sobrecarga de funciones o métodos por nombre**. Si se definen dos funciones con el mismo nombre, la segunda reemplaza silenciosamente a la primera, sin error de ningún tipo:

```python
def f1(x):
    print(x)

def f1(x, y):   # reemplaza a la anterior
    print(x)

f1(4)     # ⇒ Error (la f1 de un parámetro ya no existe)
f1(4, 5)  # ⇒ 4
```

> *"Python no permite sobrecarga de funciones, pero sí cuenta con operadores sobrecargados."*
> — Slides - Sistema de tipos - Python y TypeScript (p. 18)

La sobrecarga de **operadores** sí existe, implementada a través de *dunder methods* (`__add__`, `__mul__`, etc.). La resolución de qué `__add__` llamar se hace en **tiempo de ejecución**, sin ninguna declaración de tipos:

```python
class Vector:
    def __init__(self, x): self.x = x
    def __add__(self, other): return Vector(self.x + other.x)

v1 = Vector(1)
v2 = Vector(2)
v3 = v1 + v2  # Python llama v1.__add__(v2) en runtime
              # No hay verificación estática de tipos
```

La razón por la que Python no necesita sobrecarga de funciones es que su tipado dinámico (duck typing) ya permite que **cualquier función acepte cualquier tipo de argumento** — la "sobrecarga" está implícita en la ausencia de restricciones de tipo sobre los parámetros.

**Dureza respecto a la sobrecarga**:
- Para **funciones/métodos**: la característica no existe → no hay resolución que evaluar, pero el comportamiento (reemplazo silencioso) es extremadamente blando en cuanto a detección de errores.
- Para **operadores**: blanda — resolución dinámica en runtime, sin declaración de tipos, sin verificación estática.

---

### Resumen comparativo

| Aspecto | Pascal estándar | Java | Python |
|---|---|---|---|
| ¿Permite sobrecarga de métodos? | ❌ No | ✅ Sí | ❌ No (solo operadores) |
| Momento de resolución | N/A | Compilación (estático) | Runtime (para operadores) |
| Coerciones en resolución | N/A (solo built-in) | Sí (widening implícito) | N/A (duck typing) |
| Detección de ambigüedades | No aplica | En compilación | No aplica |
| **Dureza respecto a sobrecarga** | **Máxima** (restricción total) | **Alta** (estática, con leve blandura por widening) | **Blanda** (no hay verificación estática; reemplazo silencioso) |

La diferencia conceptual más importante: Pascal y Python no tienen sobrecarga de funciones, pero por razones opuestas. Pascal es restrictivo porque su sistema de tipos estático exigiría mecanismos complejos de resolución. Python no la necesita porque su sistema dinámico ya provee la flexibilidad que la sobrecarga pretendería aportar. Java, con tipado estático y sobrecarga explícita, tiene el sistema más sofisticado y más duro en su resolución.
