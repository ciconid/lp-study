## Pregunta 27
> ¿Por qué no tendría sentido el uso de polimorfismo paramétrico en un lenguaje como Python? ¿Lo mismo vale para cualquier lenguaje con tipado dinámico?

---

## Respuesta

### Qué es el polimorfismo paramétrico y para qué sirve

El polimorfismo paramétrico es una forma de **polimorfismo universal real** que permite definir una única función o tipo de dato que trabaja con cualquier tipo concreto pasado como parámetro. Cada instanciación concreta se verifica en forma independiente.

> *"Parametric polymorphism is provided by a subprogram that takes generic parameters that are used in type expressions that describe the types of the parameters of the subprogram. Different instantiations of such subprograms can be given different generic parameters, producing subprograms that take different types of parameters. Parametric definitions of subprograms all behave the same."*
> — Sebesta, Concepts of Programming Languages (p. 656)

Su valor principal reside en que permite combinar **flexibilidad** (una sola definición sirve para múltiples tipos) con **seguridad estática** (el compilador verifica que los tipos sean correctos en cada instanciación, antes de ejecutar el programa).

> *"For many simple abstractions, classes that do not rely on hierarchies and run-time binding provide a simpler and more efficient alternative. Furthermore, where no run-time resolution is needed, generic programming relying on (compile-time) parametric polymorphism is a better behaved and more efficient approach."*
> — Sebesta, Concepts of Programming Languages (p. 811)

---

### Por qué no tiene sentido en Python

Python tiene **tipado dinámico**: los tipos de las variables y de los parámetros formales no se declaran ni se verifican en tiempo de compilación, sino en tiempo de ejecución.

Esto tiene una consecuencia directa: **en Python, cualquier función ya es naturalmente "genérica"** — acepta cualquier tipo de argumento por defecto. No hace falta parametrizar el tipo porque el lenguaje simplemente no lo exige.

> *"A more general kind of polymorphism is provided by the methods of Python and Ruby. Recall that variables in these languages do not have types, so formal parameters do not have types. Therefore, a method will work for any type of actual parameter, as long as the operators used on the formal parameters in the method are defined."*
> — Sebesta, Concepts of Programming Languages (p. 656)

Este es precisamente el mecanismo de **duck typing**: lo relevante no es el tipo del objeto sino que tenga los atributos y métodos requeridos.

> *"Python utiliza el concepto de duck typing para realizar parte de los chequeos de tipos. Al usar duck typing, los atributos y métodos de un objeto resultan más relevantes que el tipo del objeto."*
> — Slides - Sistema de tipos - Python y TypeScript (p. 22)

```python
# Esta función en Python ya es "paramétrica" sin declararlo:
def primero(coleccion):
    return coleccion[0]  # funciona con lista, tupla, string, etc.

primero([1, 2, 3])    # ⇒ 1
primero((10, 20))     # ⇒ 10
primero("hola")       # ⇒ 'h'
```

Si se intentara trasladar la sintaxis de generics de Java a Python, el resultado sería vacuo: el parámetro de tipo `T` no añadiría ninguna verificación estática (porque el lenguaje no verifica tipos antes de ejecutar), ni restricción semántica (porque ya se acepta cualquier tipo).

```java
// En Java, sin generics, se pierde type safety:
List lista = new ArrayList();  // acepta Object
lista.add("hola");
Integer x = (Integer) lista.get(0);  // error en runtime

// Con generics, el compilador detecta el error antes:
List<Integer> lista = new ArrayList<>();
lista.add("hola");  // → Error en compilación
```

En Python esto no aplica: la lista ya puede contener objetos de cualquier tipo y los errores se detectan en runtime de todas formas. El polimorfismo paramétrico no agrega ningún beneficio que Python no tenga ya a través del duck typing.

> *"Python brinda herencia simple y múltiple. Por otro lado, no soporta polimorfismo paramétrico."*
> — Slides - Sistema de tipos - Python y TypeScript (p. 20)

En síntesis: **el polimorfismo paramétrico existe para darle al programador flexibilidad de tipos en lenguajes donde dicha flexibilidad está restringida por el tipado estático**. En Python, esa restricción no existe, así que el mecanismo sería redundante.

---

### ¿Lo mismo vale para cualquier lenguaje con tipado dinámico?

En general, **sí, el argumento central se mantiene para cualquier lenguaje puramente dinámico**: si los tipos nunca se declaran ni se verifican antes de la ejecución, una función ya acepta cualquier tipo por naturaleza, y no hay verificación estática que el polimorfismo paramétrico pueda aportar.

Sin embargo, el análisis se matiza en los siguientes casos:

#### Caso 1: tipado gradual (*gradual typing*)

Algunos lenguajes permiten optar por anotaciones de tipo en partes del código, mientras el resto permanece dinámico. En esos contextos, el polimorfismo paramétrico **sí tiene sentido** para las partes tipadas estáticamente.

El propio Python ilustra esto: el módulo `typing` (PEP 484) introduce `TypeVar` para anotar funciones genéricas, y herramientas como `mypy` verifican esas anotaciones estáticamente. Pero esto es una capa optativa sobre el lenguaje, no parte del sistema de tipos en ejecución.

```python
from typing import TypeVar, List

T = TypeVar('T')

def primero(coleccion: List[T]) -> T:
    return coleccion[0]
# mypy puede verificar: primero([1,2,3]) retorna int, etc.
```

Aquí el polimorfismo paramétrico tiene sentido, pero solo en la medida en que se usa tipado estático *dentro* del lenguaje dinámico.

#### Caso 2: lenguajes dinámicos sin duck typing universal

Si un lenguaje dinámico restringe los tipos que acepta una función por algún criterio distinto al tipado estático (por ejemplo, mediante contratos o protocolos), podría aparecer una necesidad de parametrización. Pero en la práctica, la mayoría de los lenguajes dinámicos usan duck typing o algo equivalente.

#### Conclusión

> El polimorfismo paramétrico surge como respuesta a una limitación del tipado estático: la necesidad de expresar "esta función funciona para cualquier tipo T, y el compilador lo verifica". En un lenguaje con tipado puramente dinámico, esa limitación no existe — toda función es implícitamente genérica. El polimorfismo paramétrico no añade expresividad nueva; **su valor es únicamente recuperar flexibilidad dentro de las restricciones del tipado estático**.

> *"Los lenguajes pueden aportar desde su definición diferentes grados de polimorfismo. Pueden tener todos los niveles, algunos o solo uno."*
> — 06-SistemaDeTipos (p. 3)

| Lenguaje | Tipado | ¿Polimorfismo paramétrico útil? | ¿Por qué? |
|---|---|---|---|
| Java | Estático | ✅ Sí — fundamental | Sin él, se pierde type safety o hay que duplicar código |
| Python | Dinámico | ⚠️ Solo con gradual typing (`typing`/`mypy`) | Ya hay flexibilidad total via duck typing |
| Ruby | Dinámico | ⚠️ Similar a Python | Duck typing implícito |
| TypeScript | Estático (sobre JS) | ✅ Sí | Tiene generics para las partes tipadas |
| Lenguaje puro dinámico (sin gradual typing) | Dinámico | ❌ Redundante | Cualquier función ya acepta cualquier tipo |
