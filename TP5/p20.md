## Pregunta 20
> Realice un análisis comparativo de la dureza del sistema de tipos de JavaScript y Python respecto a expresiones mixtas.

## Respuesta

### 1. Noción de dureza respecto a expresiones mixtas

Una **expresión mixta** es aquella que combina operandos de tipos distintos en una misma operación. El grado de dureza del sistema de tipos de un lenguaje respecto a esta característica está determinado por cuánto le exige al programador que explicite las conversiones de tipo, versus cuánto el lenguaje resuelve implícitamente.

> *"Un lenguaje más duro exige que el programador establezca qué operación es la que quiere aplicar, mientras que en uno más débil las transformaciones de tipo ocurren de manera implícita para poder aplicar el operador sobre tipos no esperados."*
> — 06-SistemaDeTipos (p. 2)

---

### 2. Python frente a expresiones mixtas

Python es un lenguaje de **tipado dinámico y fuertemente tipado**. Si bien no declara tipos estáticamente, sí es estricto respecto a qué combinaciones de tipos son válidas en cada operación.

> *"Python es un lenguaje interpretado con tipado dinámico, fuertemente tipado."*
> — Slides - Sistema de tipos - Python y TypeScript (p. 5)

Python admite una **franja estrecha de coerciones implícitas**, limitada principalmente al ensanchamiento dentro de la jerarquía numérica:

- `int` → `float` → `complex` (conversiones expansoras)
- `bool` se comporta como `int` (`True = 1`, `False = 0`)

> *"Para tipos numéricos, Python realiza conversiones hacia tipos más generales: un entero se convierte a real o un real a complejo. True y False se comportan como 1 y 0 si es necesario."*
> — Slides - Sistema de tipos - Python y TypeScript (p. 15)

Fuera de ese conjunto acotado, Python **lanza `TypeError`** sin intentar ninguna coerción:

```python
>> 1 + 2          # ⇒ 3        (int + int)
>> 3.14 + 1       # ⇒ 4.14     (float + int, coerción expansora)
>> True + True    # ⇒ 2        (bool → int)

>> "5" + 1        # ⇒ TypeError: can only concatenate str (not "int") to str
>> "Hola" + False # ⇒ TypeError
>> None + 5       # ⇒ TypeError
>> None + "hola"  # ⇒ TypeError
>> [1, 2, 3] + "5"# ⇒ TypeError
```

El lenguaje detecta estos errores en **tiempo de ejecución**, pero los detecta: no los ignora ni produce resultados silenciosamente incorrectos. Esto lo hace compatible con la propiedad de **fuertemente tipado**.

### 3. JavaScript frente a expresiones mixtas

JavaScript es un lenguaje de **tipado dinámico y no fuertemente tipado**. Su diseño prioriza evitar que el programa se detenga por errores de tipo, por lo que aplica **coerciones implícitas muy amplias** ante cualquier combinación de tipos incompatibles.

> *"El objetivo es evitar que el programa se detenga por un error de tipos, para lo cual los operandos mutan hacia el tipo de dato que manda el contexto."*
> — Slides - Sistema de tipos - Python y TypeScript (p. 38)

Las reglas varían según el operador:

- Con `+`: si alguno de los operandos es string, el otro se convierte a string (concatenación).
- Con `-`, `*`, `/`, `%`: cualquier operando no numérico se convierte a número.
- Cuando la conversión a número no tiene sentido (e.g., `"hola" → NaN`), el resultado es `NaN` — un valor de tipo `number`, no un error.

```javascript
>> "5" + 1          // ⇒ "51"        (número → string)
>> "Hola" + false   // ⇒ "Holafalse" (boolean → string)
>> [1,2,3] + [4,5,6]// ⇒ "1,2,34,5,6" (arrays → strings)
>> [1,2,3] + "5"    // ⇒ "1,2,35"   (array → string)
>> "42" - 1         // ⇒ 41          (string → número)
>> "hola" - 1       // ⇒ NaN         (sin error, resultado inválido)
>> NaN + "hola"     // ⇒ "NaNhola"
>> NaN + [1, 2]     // ⇒ "NaN1,2"
```

> Comparación directa para el operador `+`:
>
> | Expresión              | Python              | JavaScript          |
> |------------------------|---------------------|---------------------|
> | `[1,2,3] + [4,5,6]`    | `[1, 2, 3, 4, 5, 6]`| `"1,2,34,5,6"`      |
> | `[1,2,3] + "5"`        | `TypeError`         | `"1,2,35"`          |
> | `"Hola" + False/false` | `TypeError`         | `"Holafalse"`       |
> | `"5" + 1`              | `TypeError`         | `"51"`              |
> | `None/NaN + 5`         | `TypeError`         | `NaN`               |
> | `None/NaN + "hola"`    | `TypeError`         | `"NaNhola"`         |
>
> — Slides - Sistema de tipos - Python y TypeScript (p. 17)

### 4. Análisis comparativo

| Criterio                                | Python                            | JavaScript                          |
|-----------------------------------------|-----------------------------------|-------------------------------------|
| Coerciones implícitas permitidas        | Solo ensanchamiento numérico      | Amplias (casi cualquier combinación)|
| Respuesta ante tipos incompatibles      | `TypeError` en ejecución          | Coerción silenciosa o `NaN`         |
| Detecta errores de tipo en expr. mixtas | Sí (fuertemente tipado)           | No (no fuertemente tipado)          |
| Dureza respecto a expresiones mixtas    | **Alta**                          | **Baja**                            |

**Python** tiene un sistema de tipos **significativamente más duro** que JavaScript respecto a expresiones mixtas. Acepta solo coerciones numéricas expansoras (seguras) y rechaza cualquier otra combinación de tipos con un error explícito. Esto lo hace más predecible y confiable.

**JavaScript** tiene un sistema de tipos de **baja dureza**: convierte implícitamente casi cualquier combinación de tipos para evitar errores en tiempo de ejecución. Esto produce comportamientos sorpresivos y errores silenciosos difíciles de detectar.

> *"Las conversiones limitantes se consideran inseguras, porque se pierde información que no puede recuperarse."*
> — 06-SistemaDeTipos (p. 2)

Muchas de las coerciones de JavaScript son precisamente de carácter limitante — por ejemplo, convertir el string `"hola"` a número produce `NaN`, con pérdida total de información —, lo que contribuye a la baja seguridad del sistema de tipos.

Por estas razones, TypeScript surge como una capa sobre JavaScript que restringe estas coerciones en tiempo de compilación, elevando la dureza del sistema de tipos.

> *"TypeScript busca mantener la semántica de JavaScript, pero hace controles para reconocer errores de tipos que en JavaScript no se reconocen."*
> — Slides - Sistema de tipos - Python y TypeScript (p. 28)
